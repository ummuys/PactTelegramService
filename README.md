# Описание архитектурных решений

1) #### Передача задач в сессию

   Сначала была реализована идея "для каждой задачи создается отдельный клиент". Данный подход имел в себе следующий критический минус: постоянное создание клиента для новой задачи тратило довольно много ресурсов и времени, от чего даже самые простые задачи выполнянились намного дольше, чем должны были бы.

   После этого я принял решение реализовать идею "1 сессия на все задачи". Реализацию этой идеи вы можете в найти директории services/telegram/internal/tgapi/ в файлах:


   1) session_commands.go - файл, где есть общий интрейфейс cmd (общий вид задачи) и n структур (доступные задачи для сессии), которые реализуют этот интерфейс
   2) session_methods.go (с 83 строчки) - файл, где после авторизации задачи передаются через канал. После они запускаются через метод run().
2) #### Разбиение кода на пакеты

   Считаю это отличной практикой во время разработки проекта. Основные пакеты:


   - Services - все сервисы
   - Telegram - сервис по работе с API телеграма. Также в ней содержится:
     - adapter - hander для grpc. Обрабатываются входящие запросы и валидируются ошибки
     - services - слой бизнес-логики. Вызов нужных комманд из интерфейсов SessionManager и Session
     - tgapi - слой работы с API телеграм.
     - errs - пакет с ошибками для валидации.
     - repository - подготовленный пакет для будущей реализации сохранений сессий и восстановления их при перезапуске системы
